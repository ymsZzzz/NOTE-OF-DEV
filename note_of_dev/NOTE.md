# 嵌入式开发笔记

## 组件库

### esp_websocket_client

- 现象描述：在使用ESP官方组件库提供的esp_websocket_client库时，发现一个BUG：在多线程并发访问的环境下特别是ESP32 S3的双核架构中，例如一个任务负责向服务器端发送数据，另一个默认websocket任务在ondata事件回调中读取接受数据时。如果网络出现波动延迟，导致底层网络数据接收和发送超时，则会触发erro事件。接收线程和发送线程同时报错并触发erro事件时，都会调用该组件库的同一个方法。即中断websocket连接的方法。在该方法中会断开底层Tcp连接并释放协议栈资源。但是这个方法并没有做任何保护机制（锁），导致多线程并发访问时会发生资源竞争，最终导致协议栈资源在同一个方法中重复释放，最终导致内存报错，整个程序崩溃。

**解决办法：在发送线程中所调用的发送接口中，注释掉多线程并发访问可能会导致程序崩溃中断连接方法。即发送失败不做清理资源的操作，而是通过触发erro事件，并在应用层的事件回调处理中再调用断开连接的api来清理资源，并重新建立连接。直接避免接收线程和发送线程的并发访问。**



- 现象描述：同样的在同时两个线程并发，一个接收数据，一个发送数据。会随机性的，不确定的导致内存访问空指针错误，而导致程序崩溃。崩溃为止在send接口底层调用ssl重新认证的方法中。

**解决办法：进入menuconfig设置，关闭websocket关于SSL重新认证的配置选项即可。**



## ADC

- 分压选用**精密电阻**，测量点并联合适大小电容。
- 目前为止使用过的滤波算法：**多次采样，去掉最低值，最高值，取平均值；窗口滑动滤波**；读取的电压多次消抖，锁止。
- 一般电池电压随电量变化的曲线是**非线性**的，需要根据实际测量得出电池特性曲线，再根据数据曲线做映射处理得到电量。



## 网络协议

### websocket

- 背景：ESP32 S3作为客户端与服务器建立websocket长连接，并进行流式音频数据传输。

  问题：由于网络延迟问题，服务器端，客户端如果不约定进行流控，会导致客户端（MCU）接收数据缓冲区不够用，直接导致丢包，音频播放卡顿不顺畅。

  解决方案：与服务器约定好流控策略，使用令牌桶策略。根据客户端消费资源（流式音频数据）速度，来下发令牌数量，服务器端根据令牌数量进行音频包的发送。



## 工厂模式

### esp32 s3

- 工厂固件和正式固件理论上menuconfig应该配置一致，并且分区表应该严格保持一致，这会直接影响到调用切换分区API时的正确性。
- 在首次烧录固件时，不烧录otadata分区，原因：bootloader在启动时会检测otadata中的内容，如果没有为空，则会默认启动app分区中的factory分区。
- 在结束工厂模式之后，调用api切换到ota0分区中的正式固件。
- 在首次运行正式固件时，需要在程序最前端检查分区切换验证，标记为正常固件，否则在断电后会进行分区回滚，回滚到factory固件。
- 并且更改分区表之后（加入factory分区），无法再使用idf烧录，原因：默认将正式固件烧录到factory分区导致大小不匹配。只能自定义烧录地址后烧录。



## CAT1模组

### 移远（EC800系列）

- 建立于MCU的通信，推荐使用PPP拨号的形式，使所有的协议交给MCU来做，更加灵活且支持并发。
- PPP拨号一般来说有UART和USB两种方式，但是大多数情况推荐使用UART，因为USB需要专门的驱动才行，移远目前只支持windows和linux驱动。
- 定位信息的获取，获取之后的经纬度一般是国际通用格式，但是在国内开发需要注意，国内法律法规规定国内地图软件不能使用国际通用的参考坐标系，直接使用国内软件查询定位信息，得到位置与实际往往相差5-600米。需要经过一次坐标系转换。



## 低功耗

### 移远（EC800系列）

1. AT标准开发方式（UART连接）中，控制模组**进入休眠状态**的前提条件，确保提前执行**AT+QSCLK = 1**; 需要是模组进入休眠时再**拉高DTR引脚**。 按照官方开发资料表明：
   ![image](https://img2024.cnblogs.com/blog/3746389/202601/3746389-20260126104206480-294274764.png)

   

2. 实测表明：执行完上述两步操作之后，模组也并**不会立刻进入休眠状态**，仍然可以回复AT指令，并进入PPP拨号连接。但是随后也会断开网络连接，经过研究发现该现象与模组内部机制设计吻合，属于正常现象。进入休眠状态由模组内部任务决定，并不直接由**AT+QSCLK = 1** 和 **拉高DTR引脚** 直接决定，主控只能控制模组获得进入休眠状态的“门票”，**具体进入休眠状态的时机由模组内部决定**。并且据实测表明，模组进入休眠状态之后，发送连续发送AT指令也是可以回复的，可以将模组短暂的唤醒。
   ![image](https://img2024.cnblogs.com/blog/3746389/202601/3746389-20260126104142947-230852980.png)

3. 因此在针对低功耗方案开发过程中，应该明确进入低功耗的时机与前置操作。控制模组进入休眠状态之前，主控应该主动断开PPP连接，并且不再发送任何额外AT指令，再拉高DTR引脚（提前执行**AT+QSCLK = 1**），等待模组自行进入休眠状态。 需要唤醒时，主控首先拉低DTR引脚，随后继续发送AT指令确保模组成功被唤醒之后，再重新建立PPP拨号。
