# AFE算法内存占用与初始化顺序策略

## 问题背景

在 AFE 相关能力（如唤醒词、回声消除、降噪、人声处理）配置中，常见内存策略为：

1. 优先使用 RAM
2. RAM 与 PSRAM 平衡
3. 优先使用 PSRAM

即使选择“优先使用 PSRAM”，工程上仍可能观察到 AFE 占用一定比例的 RAM。当前结论基于实测经验，具体与 IDF 版本、组件实现、运行时内存状态相关，需在项目内补充版本信息。

## 实测现象

- AFE 初始化阶段会占用部分 RAM。
- 占用量不是固定值，存在随系统当下可用 RAM 变化的特征。
- 经验示例：
  - 若当前剩余 RAM 约 100 KB，AFE 可能占用约 50 KB。
  - 若当前剩余 RAM 约 200 KB，AFE 可能占用约 100 KB。

以上示例用于说明“动态占用趋势”，不应作为固定比例规则。

## 风险场景

若先初始化 AFE，再初始化 Wi-Fi/Bluetooth 等对 RAM 依赖更强的模块，可能导致后续模块初始化失败，表现为：

- 初始化直接失败
- 初始化成功率不稳定
- 不同启动批次内存行为不一致

## 根因判断（工程推断）

基于现象的工程推断：AFE 在初始化时会按当时内存状态申请工作内存，即便策略偏向 PSRAM，仍会保留一定 RAM 使用；当 AFE 先启动并占用较多 RAM 时，后续必须使用 RAM 的模块会被挤压可用空间。

该结论为实测推断，不代表官方文档中的确定性机制描述。

## 推荐初始化顺序

默认建议采用以下顺序：

1. 初始化基础系统资源（时钟、任务、队列等）
2. 初始化 Wi-Fi/Bluetooth 等 RAM 敏感模块
3. 最后初始化 AFE 模型与相关音频处理链路

核心目标是优先保障“必须依赖 RAM”的模块可用性，再引入 AFE 的弹性占用行为。

## 规避策略 A：延迟 AFE 初始化（默认）

将 AFE 初始化延迟到 Wi-Fi/Bluetooth 初始化完成后执行。

优点：

- 实现简单，侵入性低
- 行为稳定，易于维护

注意点：

- 需要在状态机中明确“音频链路可用时机”
- 避免业务层在 AFE 未就绪前触发依赖调用

## 规避策略 B：初始化前 RAM 临时占位

在 AFE 初始化前，先根据系统可用 RAM 申请一段临时内存进行“占位”；待 AFE 初始化完成后，再安全释放该占位内存，以限制 AFE 在初始化阶段的 RAM 抢占空间。

优点：

- 可主动收敛 AFE 初始化期 RAM 占用上限

风险与约束：

- 实现复杂度更高，需要严格的申请/释放配对
- 占位大小设置不当会导致 AFE 初始化失败
- 异常路径必须保证内存可回收，避免引入泄漏或碎片化
- 建议仅在策略 A 无法满足目标时使用

## 排查与验证清单

### 初始化前后 RAM 观测点

- 记录 AFE 初始化前可用 RAM
- 记录 AFE 初始化后可用 RAM
- 记录 Wi-Fi/Bluetooth 初始化前后可用 RAM
- 对比不同启动批次的波动区间

### 失败复现条件

- 固定构建配置，重复冷启动/热启动
- 控制网络与蓝牙状态一致，减少外部变量
- 分别验证“AFE 先初始化”与“AFE 后初始化”的差异

### 回归检查项

1. Wi-Fi 初始化稳定通过
2. Bluetooth 初始化稳定通过
3. AFE 初始化稳定通过
4. 长时间运行后无内存异常（泄漏、碎片恶化、随机失败）
5. 多轮重启后行为一致
